# 1. Golden Rule for the Merged Plan

We will not follow Sprint numbers blindly anymore.

Instead, we follow **dependency order**.

Your TaskList has tasks grouped as Sprint 2 → Sprint 6, but to build the strongest application, some tasks must be slightly re-orchestrated. For example, correlation is CRITICAL, but correlation cannot be strong unless import, staging, canonical data, and mapping are stable first.

# 5. What We Must Not Do

To keep the application strong, we must avoid these traps:

| Trap                               | Why Dangerous                                       |
| ---------------------------------- | --------------------------------------------------- |
| Jumping to ML now                  | No stable import/mapping/correlation foundation yet |
| Building only dashboards           | Product becomes BI-only, against vision             |
| Hard-coding steel route            | Breaks generic manufacturing strategy               |
| Keeping endpoint business logic    | Hard to maintain and test                           |
| Skipping raw/staging               | No audit, no replay, no customer trust              |
| Creating risk scores manually only | Not real prediction                                 |
| Correlation only in frontend/Qlik  | Intelligence must stay in backend/API               |
| Building microservices too early   | Slows you down before product-market proof          |
