0. MASTER INTENT
----------------
PlantProcess IQ is a generic manufacturing quality-intelligence and prediction platform. It is not only a BI dashboard and not a replacement for MES, Level 2, SCADA, PLC, ERP, QMS or historian systems. It is an intelligence layer that reads from existing systems, connects fragmented production/process/inspection/quality data, builds product or batch genealogy, identifies correlations between process parameters and quality defects, predicts quality risk, and supports engineers with evidence-based investigation and recommendations.

Karim's original example is the steel route:
EAF / Ladle Furnace -> Caster -> Hot Strip Mill -> Inspection / Quality -> Final Decision.
The same design must also work for aluminum, pharma, tire, paper, automotive, chemicals, food and other manufacturing industries by using a domain-neutral canonical data model and industry-specific configuration templates.

Core rule:
Build the generic manufacturing core first. Steel-specific terms such as Heat, Ladle, Cast, Slab, Coil, Rail, Section, Mould, Tundish or Powder must be configured metadata, not hard-coded architecture.

1. PRODUCT POSITIONING
----------------------
Recommended product name: PlantProcess IQ.
Alternative names:
- SteelProcess IQ for steel-focused demonstration.
- PlantQuality IQ for cross-industry positioning.
- Quality Genealogy AI for traceability-focused positioning.
- Process-to-Quality AI for analytics-focused positioning.

Recommended product statement:
PlantProcess IQ connects production, process, inspection and quality data from existing MES, Level 2, SCADA/PLC/OPC, historian, laboratory, inspection, downtime and ERP systems. It builds material or batch genealogy, detects data-quality problems, identifies process-to-defect correlations, predicts quality-risk probability, and provides dashboards, investigation reports and AI-assisted explanations for production, quality and process teams.

Recommended commercial statement:
Many plants already collect huge amounts of data, but the data is fragmented and difficult to use for root-cause analysis. PlantProcess IQ converts fragmented data into connected quality intelligence so teams can investigate defects faster, predict quality risk earlier, and reduce scrap, rework, downgrade, claims and manual analysis time.

5. CANONICAL DATA MODEL - DETAILED ENTITIES
-------------------------------------------
The following entities should form the generic product core.

5.1 Plant and structure
- Tenant: customer/company boundary.
- Site: plant or factory.
- Area: production area such as melt shop, caster, rolling, finishing, lab.
- Line: specific production line.
- Operation: process operation such as melting, refining, casting, rolling, curing, coating, filling.
- Equipment: physical equipment such as furnace, ladle, mould, mill stand, reactor, press, machine, inspection device.
- EquipmentGroup: group of similar assets for comparison.
- Crew/Shift: operational shift and crew metadata.

5.2 Production and material
- WorkOrder: customer/production order or batch order.
- ProductFamily: steel grade family, pharma product, tire type, paper grade, component family.
- MaterialUnit: any traceable physical unit: heat, batch, cast, slab, billet, bloom, coil, rail, section, roll, lot, tire, part.
- MaterialAlias: source-system-specific IDs, because one material can have multiple IDs across MES/L2/ERP/QMS.
- GenealogyEdge: parent-child relationship between material units.
- ProcessStepExecution: actual execution of a process step for a material unit.
- Route: expected process path.
- RouteStep: expected sequence of operations.

5.3 Process parameters
- ParameterDefinition: parameter metadata: name, code, unit, type, min/max, category, industry alias.
- ParameterObservation: actual measured or calculated value with timestamp, material, equipment, source and quality flag.
- ParameterWindow: expected/safe range by product family, route, operation and equipment.
- AggregatedFeature: engineered feature such as average casting speed, max temperature, duration above limit, standard deviation, fluctuation index.

5.4 Quality and inspection
- QualityEvent: defect, deviation, rejection, downgrade, claim, hold, warning.
- DefectCatalog: standardized defect type and category.
- InspectionResult: result from visual, NDT, lab, automatic measurement, camera, manual test.
- Sample: physical or laboratory sample linked to material and process step.
- SampleResult: chemistry, mechanical, dimensional or quality result.
- QualityDecision: accepted, downgraded, rejected, reworked, on hold, customer claim.

5.5 Time loss and events
- DowntimeEvent: downtime, delay, breakout, crack, stoppage, maintenance stop.
- ProcessEvent: notable event such as breakout, caster crack, speed drop, roll change, mold powder change, furnace delay.
- EventCause: operator-entered or system-derived cause.

5.6 Analytics and ML
- DataQualityIssue: missing data, duplicate event, invalid timestamp, broken genealogy, suspicious value.
- CorrelationResult: correlation/risk relationship found between features and quality outcome.
- ModelRun: metadata for trained ML model.
- RiskScore: predicted probability and classification for material/batch.
- Explanation: feature importance / SHAP-style explanation / natural-language reason.
- Recommendation: action or investigation suggestion.
- InvestigationReport: generated report for quality/process review.

10. MACHINE LEARNING AND CORRELATION STRATEGY
---------------------------------------------
The ML path should mature in stages.

Stage A - Data readiness and quality:
- check missing values;
- check duplicate records;
- validate timestamps;
- validate genealogy links;
- standardize defect labels;
- normalize units;
- compare source systems.

Stage B - Descriptive analytics:
- defect rate by grade/product family;
- defect rate by equipment;
- defect rate by route;
- defect rate by crew/shift;
- defect rate by parameter ranges;
- downtime/delay impact on quality;
- trend analysis by time.

Stage C - Correlation / association:
- Pearson/Spearman correlation for numeric values;
- Cramer's V / Chi-square for categorical associations;
- ANOVA/Kruskal-Wallis for group differences;
- lift analysis for defect probability by parameter range;
- mutual information for nonlinear relationships.

Stage D - Baseline prediction:
- logistic regression for simple interpretable classification;
- decision tree for simple rules;
- random forest for robust baseline;
- gradient boosting / XGBoost / LightGBM later for higher performance;
- isolation forest or one-class models for anomaly detection;
- clustering for process pattern grouping.

Stage E - Explainability:
- feature importance;
- permutation importance;
- SHAP-style explanations later;
- natural-language explanation templates;
- confidence score and warning that correlation does not prove root cause.

Stage F - Recommendation assistant:
- first version: rule-based recommendations from process engineers and correlation findings;
- second version: model-driven risk contributors;
- third version: AI assistant summarizing likely contributors and suggested investigation steps;
- do not automatically send control commands to process equipment.

Important scientific rule:
The product must say "correlated with", "risk indicator", "suspected contributor" or "recommended investigation", not "guaranteed root cause", unless validated by process experts.

PlantProcess IQ

Technical Blueprint for a Generic Manufacturing Quality Intelligence & Prediction Platform

Prepared for: Karim Gamal Elsayed
Version 1.0 | 2026

1. Executive Technical Summary

PlantProcess IQ is designed as a generic intelligence layer above existing MES, Level 2, historian, quality, laboratory, inspection, downtime and ERP systems. It does not replace MES, SCADA or BI. It connects process and production data end-to-end, builds material genealogy, identifies correlations between process parameters and quality outcomes, predicts quality risk, and explains suspected contributors in a dashboard and AI-assisted report.

Core design principle Build a domain-neutral manufacturing data model first. Industry-specific wording such as heat, coil, batch, tire, roll, lot, paper reel or aluminum billet must be configured as metadata, not hard-coded logic.
--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

2. Product Objectives

Connect fragmented production, process, inspection and quality data across the plant.

Normalize data into a canonical manufacturing model that works for process and discrete manufacturing.

Build genealogy from raw material, batch or order through process steps to final product and quality decision.

Detect missing, duplicated, delayed or inconsistent production/quality records.

Run statistical correlation and machine-learning models to identify risk contributors.

Predict quality risk before the next process step and provide explainable recommendations.

Offer dashboards in Qlik/Power BI for early MVP, and a React-based web application for a scalable product.

3. Recommended Product Architecture

Figure 1 - Reference architecture for a generic manufacturing quality intelligence platform.

The architecture should separate the product into layers. This avoids building a tool that only works for one plant or one industry. Each layer has a clear responsibility and can be replaced or extended later.

Layer                | Purpose                                                                                                                             | Recommended Technology                                                                                                 
-------------------- | ----------------------------------------------------------------------------------------------------------------------------------- | -----------------------------------------------------------------------------------------------------------------------
Data connector layer | Read from MES, L2, PLC/OPC exports, historian, lab, inspection, downtime, ERP and BI data sources.                                  | C#/.NET connectors, Python ingestion scripts, ODBC/JDBC, REST APIs, CSV/Excel import, OPC UA snapshots where permitted.
Staging layer        | Store raw snapshots exactly as received for auditability and repeatable testing.                                                    | PostgreSQL or SQL Server; object storage/parquet later for large data.                                                 
Canonical data layer | Transform plant-specific source data into generic entities: material, operation, equipment, parameter, quality event and genealogy. | PostgreSQL/SQL Server relational model with JSONB/JSON columns for flexible attributes.                                
Analytics layer      | Run data quality rules, correlation, anomaly detection and feature engineering.                                                     | Python/scikit-learn, SQL analytics, optional ML.NET for .NET-native models.

4. Database Strategy

4.1 Recommended MVP database

For the first version, use PostgreSQL as the product database because it is open, strong for relational modeling, supports declarative table partitioning, and provides JSON/JSONB types for flexible attributes. PostgreSQL partitioning is useful for large time-based observation tables, and JSONB is useful for plant-specific parameters while the canonical model is still evolving.

4.2 Enterprise customer database support

Support Microsoft SQL Server and Oracle as source systems because many MES/L2 environments already use them.

Keep source connectors separate from the product storage layer.

Never force a plant to change its existing MES/L2 databases.

For pilots, accept database exports if direct connection is politically or technically difficult.

For production, prefer read-only database users, API exports, or scheduled extracts approved by the customer IT/security team.

Stage                | Method                                                                             | Purpose                                                                   | Output

Data profiling       | Missing values, duplicates, timestamp gaps, distribution checks                    | Know if the customer data is usable.                                      | Data readiness report.
